🎯 MediatR Pattern Explained

Making Complex Code Simple Through Mediation

What Happens in Your Code?

Without MediatR (What You Might Write):

public class AuthController : ControllerBase { private readonly IApplicationDbContext _db; private readonly ITokenService _tokenService; private readonly ILogger<AuthController> _logger; public AuthController(IApplicationDbContext db, ITokenService tokenService, ILogger log) { _db = db; _tokenService = tokenService; _logger = logger; } [HttpPost("login")] public async Task<IActionResult> Login(LoginRequest request) { // 100 lines of logic: query user, verify password, lock account, // generate token, save to DB, log... // ALL IN THE CONTROLLER! } }

With MediatR (What You Have Now):

public class AuthController : ControllerBase { private readonly IMediator _mediator; // ← ONE dependency! public AuthController(IMediator mediator) => _mediator = mediator; [HttpPost("login")] public async Task<IActionResult> Login(LoginRequest request, CancellationToken ct) { var result = await _mediator.Send(new LoginCommand(request.Username, request.Password), ct); return Ok(result); // ← That's it! Logic is elsewhere. } }

🌍 Real-Life Analogy: Restaurant Order System

Without MediatR:

You (the customer) walk into the kitchen and tell each chef directly:

"Hey chef, I want a burger. Let me give you the beef (cooking dependency), the grill (equipment), the seasoning supplies (validator). And I'll watch while you cook it and give you instructions..."

With MediatR:

You hand a written order slip to the waiter (Mediator):

"Here's my order: one burger. Waiter, you figure out which chef, what ingredients they need, and when the burger is ready, bring it to me."

Benefits:

  • ✅ You don't need to know how to cook (the controller doesn't know implementation details)
  • ✅ Chef can change the recipe without telling customers (loose coupling)
  • ✅ Waiter ensures quality (validation, logging, error handling happens automatically)
  • ✅ Multiple people can order simultaneously without conflict (clean separation)

🔄 What Gets Replaced by Dependency Injection?

In AuthController Replaced By Reason
IApplicationDbContext IMediator Controller doesn't directly access DB anymore. Mediator routes to handler.
ITokenService IMediator Controller doesn't create tokens. Handler (via DI in Handler) does.
ILogger IMediator Handler logs internally. Controller doesn't need to.
Login() { ...100 lines... } LoginCommandHandler All logic moved to dedicated handler class. Controller just sends command.

🏗️ Where Is The Logic?

Login logic breakdown:

  • LoginCommandHandler.cs → Contains Handle() method with the actual logic
  • LoginCommand.cs → Request object (carries Username, Password)
  • AuthResponseDto → Response object (carries AccessToken, RefreshToken, etc.)

What the Handler Does (line by line):

  1. 📥 Receives LoginCommand (request) → extracts Username, Password
  2. 🔍 Queries DB for user by username (using injected IApplicationDbContext)
  3. 🔐 Verifies password with BCrypt
  4. 🚫 Checks if account is locked (failed attempts ≥ 5)
  5. 📝 Increments failed attempt counter if password wrong
  6. 🎟️ Generates access token (using injected ITokenService)
  7. 🔄 Generates refresh token
  8. 💾 Saves refresh token to DB
  9. 📢 Logs successful login
  10. 📤 Returns tokens + user info as AuthResponseDto

⚙️ How Dependency Injection Works in Handler

// In Program.cs, someone registered these: builder.Services.AddScoped<IApplicationDbContext, ApplicationDbContext>(); builder.Services.AddScoped<ITokenService, TokenService>(); builder.Services.AddMediatR(...); // Registers all handlers // When you send LoginCommand via mediator: // 1. MediatR finds LoginCommandHandler // 2. Looks at LoginCommandHandler constructor: // public LoginCommandHandler(IApplicationDbContext db, ITokenService tokenService, ILogger logger) // // 3. DI Container says: "I have these implementations!" // - IApplicationDbContext → new ApplicationDbContext() // - ITokenService → new TokenService() // - ILogger → new Logger() // // 4. Creates instance: new LoginCommandHandler(db, tokenService, logger) // 5. Calls Handle(loginCommand)

🎯 Benefits of This Pattern

1. Single Responsibility

Controller's job: "Receive HTTP request, send command, return response"

Handler's job: "Execute business logic"

2. Testability

Easy to test LoginCommandHandler independently by mocking IApplicationDbContext and ITokenService

3. Reusability

Same LoginCommandHandler can be called from Controllers, CLI, or background jobs

4. Automatic Cross-Cutting Concerns

Validation, logging, error handling happen automatically in the pipeline

5. Loose Coupling

Controller doesn't know HOW login works, just WHAT command to send

📋 Comparison: Without vs With MediatR

❌ Without MediatR
  • 🎯 Controller: 1 injection
  • ❌ Controller: 50-100 lines of logic
  • ❌ Controller tightly coupled to DB, TokenService
  • ❌ Hard to test (mocks required for controller test)
  • ❌ No automatic validation pipeline
  • ❌ Logic only accessible from controller
✅ With MediatR
  • 🎯 Controller: 1 injection (IMediator)
  • ✅ Controller: 3 lines of code
  • ✅ Controller is a simple router
  • ✅ Easy to test handler in isolation
  • ✅ Validation runs automatically
  • ✅ Logic reusable from anywhere

💡 Key Takeaway

MediatR is a "Request/Response" library that acts like a middleman.

Instead of the Controller directly calling 3-4 services and orchestrating complex logic:

  • Controller says: "I have a LoginCommand, here it is"
  • MediatR says: "I'll find someone who knows how to handle that"
  • Handler says: "I'll use my dependencies (DB, TokenService, Logger) to process this"
  • MediatR says: "Great, here's the result" and returns it to the controller

This keeps code organized, testable, and reusable. 🎉
