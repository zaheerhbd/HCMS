🎤 Interview Q&A
Q: What is new in Angular 19 / recent Angular versions?
A: The biggest shift is standalone components — you no longer need NgModules. Every component declares its own imports. 02 · Project

Folder Structure

The project follows a feature-first structure. Every feature area lives in its own folder under features/. Shared infrastructure (services, guards, interceptors) lives in core/.

src/app/ │ ├── core/ ← shared infrastructure, loaded ONCE │ ├── guards/ │ │ ├── auth.guard.ts ← checks if logged in │ │ └── role.guard.ts ← checks user role │ ├── interceptors/ │ │ ├── jwt.interceptor.ts ← attaches Bearer token to every request │ │ └── error.interceptor.ts ← shows snackbar on 500/network errors │ ├── models/ │ │ └── auth.models.ts ← TypeScript interfaces │ └── services/ │ └── auth.service.ts ← login, logout, token storage │ ├── features/ ← one folder per feature area (lazy loaded) │ ├── auth/login/ ← login page │ ├── dashboard/ │ ├── patients/ ← has its own routes file │ ├── cases/ ← has its own routes file │ ├── tasks/ │ ├── documents/ │ ├── appointments/ │ ├── reports/ │ ├── admin/ │ └── unauthorized/ │ ├── shell/app-shell/ ← main layout (sidebar + top bar + router-outlet) │ ├── app.component.ts ← root component, just has <router-outlet> ├── app.config.ts ← providers (DI root, replaces AppModule) └── app.routes.ts ← all top-level routes
🎤 Interview Q&A
Q: Why separate core/ from features/?
A: core/ contains things that are instantiated once for the whole app — services, guards, interceptors. features/ contains everything specific to a feature. This separation makes it easy to find code, avoids circular imports, and mirrors the principle of single responsibility.
03 · Architecture

Standalone Components

In Angular 19 every component in this project is standalone. That means it declares its own dependencies directly in the imports array — no NgModule in between.

What does standalone look like?

@Component({
  selector: 'app-login',
  standalone: true,           // ← this makes it standalone
  imports: [
    CommonModule,
    ReactiveFormsModule,      // ← import what THIS component needs
    MatFormFieldModule,
    MatButtonModule,
  ],
  templateUrl: './login.component.html',
})
export class LoginComponent { ... }

Old way (NgModule) vs new way (Standalone)

NgModule (old)Standalone (this project)

Where do you import dependencies?In the NgModule's imports arrayDirectly in the component's imports array
How many files per feature?Component + Module (at minimum)Just the component

🎤 Interview Q&A
Q: What is a standalone component and why use it?
A: A standalone component manages its own imports instead of relying on an NgModule. Benefits: less boilerplate (no module file to maintain), easier tree-shaking (only import what you actually use), simpler lazy loading (you can lazy-load a single component, not a whole module), and clearer ownership — you can see exactly what a component depends on just by looking at it.
04 · Architecture

Bootstrap — app.config.ts

The app starts in main.ts which calls bootstrapApplication() with an AppComponent and an appConfig object. Think of appConfig as the replacement for the old AppModule.

// main.ts
bootstrapApplication(AppComponent, appConfig)

// app.config.ts — the DI root for the whole app
export const appConfig: ApplicationConfig = {
  providers: [
    provideZoneChangeDetection({ eventCoalescing: true }),  // change detection
    provideRouter(routes, withComponentInputBinding()),       // routing
    provideHttpClient(withInterceptors([                      // HTTP + interceptors
      jwtInterceptor,
      errorInterceptor
    ])),
    provideAnimationsAsync()                                   // Material animations
  ]
};

What each provider does

ProviderWhat it sets up
provideZoneChangeDetectionEnables Zone.js-based change detection. eventCoalescing: true batches multiple DOM events into one detection cycle — better performance.
provideRouterSets up the Angular Router with our route definitions. withComponentInputBinding() allows route params to be passed as @Input() to components.
provideHttpClientSets up the HttpClient service. withInterceptors() registers the functional interceptors in order.
provideAnimationsAsyncLoads Angular Material animation support asynchronously (better initial load time than the old synchronous version).
🎤 Interview Q&A
Q: What replaced AppModule in Angular 19?
A: bootstrapApplication() with an ApplicationConfig object (defined in app.config.ts). Instead of putting imports and providers in an NgModule class, you use provider functions like provideRouter(), provideHttpClient(), and provideAnimationsAsync(). This is more tree-shakable and explicit.

06 · Architecture

Lazy Loading

Every component and route set in this app is lazy loaded. The browser only downloads the JavaScript for a feature when the user actually navigates to it. This keeps the initial bundle small and the app fast.

Two types of lazy loading used here

1 — loadComponent (single lazy component)

{
  path: 'dashboard',
  loadComponent: () =>
    import('./features/dashboard/dashboard.component')
      .then(m => m.DashboardComponent)
}

The import() is a dynamic import — a browser/JS feature. Angular's router calls this function only when the user navigates to /dashboard. Until then, that code is not downloaded.

2 — loadChildren (lazy route set)

{
  path: 'patients',
  loadChildren: () =>
    import('./features/patients/patients.routes')
      .then(m => m.PATIENTS_ROUTES)
}

Used for features with multiple sub-routes (e.g. /patients/list, /patients/:id). The router lazily loads the entire route array for that feature at once.

How the browser loads it — step by step

User visits /patients
→
Router sees loadChildren
→
Dynamic import() fires
→
Browser downloads patients chunk
→
Component renders

What you see in the Network tab

On first load you get main.js (small). When you navigate to /patients for the first time you see a new JS file like patients-routes-XXXXX.js download. On second visit — already cached, instant.

🎤 Interview Q&A
Q: What is lazy loading and why do we use it?
A: Lazy loading means code is only downloaded when it is needed. Without it, the browser would download JavaScript for every page — including pages the user may never visit. In a large app like HCMS with 8 feature areas, lazy loading can reduce the initial bundle by 60–80%, making the first page load significantly faster.

Q: What is the difference between loadComponent and loadChildren?
A: loadComponent lazily loads a single standalone component. loadChildren lazily loads a whole route configuration (an array of routes), which is useful when a feature has multiple sub-pages like list and detail views.
07 · Core Layer

Services & Dependency Injection

AuthService is the only service built so far. It handles login, logout, token storage, and exposes the authentication state.

@Injectable({ providedIn: 'root' })   // ← singleton for the whole app
export class AuthService { ... }

What providedIn: 'root' means

providedIn: 'root' tells Angular to create one single instance of this service and share it everywhere. You do not need to add it to any providers array. Angular's tree-shaker will also remove it from the bundle if nothing actually uses it.

What AuthService does

Method / PropertyWhat it does
login(request)POST to /api/auth/login, stores tokens in localStorage on success
refreshToken()POST to /api/auth/refresh, swaps old refresh token for new tokens
logout()Calls /api/auth/logout, clears localStorage, redirects to /login
getAccessToken()Returns JWT from localStorage (used by jwtInterceptor)
hasRole(role)Checks if current user has a given role (used by roleGuard + AppShell)
isAuthenticated$BehaviorSubject observable — emits true/false when auth state changes

Token storage — why localStorage?

Tokens are stored in localStorage under keys like hcms_access_token. This survives page refresh. The trade-off is XSS risk — if an attacker injects script, they can read localStorage. For a portfolio project this is acceptable; in production HCMS uses HttpOnly cookies for extra security.

🎤 Interview Q&A
Q: What is Dependency Injection in Angular?
A: DI is Angular's system for providing objects (services) to classes that need them, without those classes creating the objects themselves. You declare what you need in the constructor, and Angular's injector provides it. This makes code easier to test (you can inject a mock) and avoids duplicating instances.

Q: What is the difference between providedIn: 'root' and adding a service to a component's providers array?
A: providedIn: 'root' creates one singleton shared by the whole app. Adding to a component's providers creates a new instance scoped to that component and its children — destroyed when the component is destroyed. For AuthService we want a single instance everywhere, so we use root.
08 · Core Layer

HTTP Interceptors

Interceptors sit between your code and every HTTP request/response. Think of them as middleware for the browser. This project has two.

How they are registered (app.config.ts)

provideHttpClient(withInterceptors([jwtInterceptor, errorInterceptor]))

They run in order: jwt first, then error. Both are plain functions (not classes) — the Angular 19 way.

Interceptor 1 — jwtInterceptor

Job: Attach the JWT Bearer token to every outgoing request. If a request gets a 401 (token expired), automatically refresh and retry once.

Outgoing request
→
Has token?
→
Add Authorization header
→
Send
Response = 401
→
Call refreshToken()
→
Retry original request
→
If refresh fails → logout
export const jwtInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const token = auth.getAccessToken();

  // Clone the request — requests are immutable
  const authReq = token
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;

  return next(authReq).pipe(
    catchError((err) => {
      if (err.status === 401 && !req.url.includes('/auth/')) {
        return auth.refreshToken().pipe(
          switchMap(res => next(req.clone({
            setHeaders: { Authorization: `Bearer ${res.accessToken}` }
          })))
        );
      }
      return throwError(() => err);
    })
  );
};
Why clone the request? HttpRequest objects in Angular are immutable. You cannot modify them — you must call .clone() with the changes you want.

Interceptor 2 — errorInterceptor

Job: Show user-friendly snackbar toasts for network errors (status 0) and server errors (5xx). Lower-level errors like 401/403 are handled elsewhere.

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const snackBar = inject(MatSnackBar);

  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 0)        // no internet / server down
        snackBar.open('Unable to connect...', 'Close');
      else if (err.status >= 500)  // server crashed
        snackBar.open('A server error occurred...', 'Close');
      return throwError(() => err);  // still propagate error
    })
  );
};
🎤 Interview Q&A
Q: What is an HTTP interceptor and what can you use it for?
A: An interceptor is middleware that intercepts every HTTP request or response before it reaches your component. Common uses: adding auth headers (like we do with JWT), logging, showing loading spinners, transforming response shapes, handling errors globally, and retrying failed requests.

Q: Why do we check !req.url.includes('/auth/') in jwtInterceptor?
A: To prevent an infinite loop. If the refresh token call itself returns a 401, we don't want to try refreshing again — that would loop forever. By excluding /auth/ URLs from the retry logic, we ensure only non-auth requests trigger the refresh flow.
09 · Core Layer

Route Guards

Guards decide whether a route can be activated. Both guards in this project are plain functions — the modern Angular approach.

authGuard — is the user logged in?

export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.getAccessToken()) return true;  // has token → allow

  router.navigate(['/login']);
  return false;                               // no token → redirect to login
};

Applied to the AppShellComponent route — which means all child routes (dashboard, patients, cases…) are protected by a single guard.

roleGuard — does the user have the right role?

export const roleGuard: CanActivateFn = (route) => {
  const auth = inject(AuthService);
  const requiredRoles = route.data['roles'] ?? [];  // read from route config

  if (!auth.getAccessToken()) { router.navigate(['/login']); return false; }

  const hasRole = requiredRoles.some(r => auth.hasRole(r));
  if (!hasRole) { router.navigate(['/unauthorized']); return false; }

  return true;
};

Applied only to /admin route with data: { roles: ['Admin'] }.

Guards protect routes — not UI elements. A clever user could still see a nav link they shouldn't click. That's why AppShellComponent also filters visibleNav based on roles — double protection.
🎤 Interview Q&A
Q: What is the difference between authGuard and roleGuard?
A: authGuard only checks if you are logged in (has a token). roleGuard checks who you are — it reads the required roles from the route's data property and checks if the current user has any of them. They work together: authGuard is the bouncer at the door, roleGuard is the VIP list check inside.

Q: Why are guards plain functions now instead of classes?
A: Functional guards (introduced in Angular 14) are simpler — no class boilerplate, no implements interface, and they can use inject() to get dependencies just like components. They're also easier to unit test since you can call them as regular functions.

Models & Interfaces

TypeScript interfaces define the shape of data. They exist only at compile time — no runtime overhead.

// auth.models.ts

interface LoginRequest {
  username: string;
  password: string;
}

interface AuthResponse {       // shape of what the API returns
  accessToken: string;
  refreshToken: string;
  expiresAt: string;
  username: string;
  roles: string[];
}

interface AuthState {          // shape of local auth state
  accessToken: string | null;
  refreshToken: string | null;
  username: string | null;
  roles: string[];
  isAuthenticated: boolean;
}
🎤 Interview Q&A
Q: Interface vs class in TypeScript — when do you use each?
A: Use an interface when you just need to describe the shape of data (like an API response). They compile away completely — zero runtime cost. Use a class when you need behavior (methods), constructor logic, or dependency injection. In this project, data contracts are interfaces and injectables are classes.

A: Reactive forms are defined in the component class using FormGroup and FormControl — the form model lives in TypeScript. 14 · Advanced

Change Detection

Change detection is how Angular knows when to update the DOM. This is one of the most common interview topics.

Default strategy — Zone.js

By default Angular uses Zone.js which patches browser APIs (setTimeout, click events, HTTP calls) to detect when something might have changed. After any async event, Angular checks the whole component tree from top to bottom.

User clicks button ↓ Zone.js detects the event ↓ Angular runs change detection on EVERY component in the tree ↓ Updates DOM where needed

eventCoalescing: true (used in this project)

provideZoneChangeDetection({ eventCoalescing: true })

If multiple events fire close together (e.g. mousedown + mouseup + click), without coalescing Angular runs change detection three times. With eventCoalescing: true it batches them into one run. Better performance, especially in forms and drag interactions.

OnPush strategy (not used yet — but know it for interviews)

With ChangeDetectionStrategy.OnPush Angular skips a component unless:

  • An @Input() reference changes
  • An event originates from the component itself
  • An Observable it's subscribed to via async pipe emits

OnPush components are much faster in large apps. Phase 2+ of HCMS will use OnPush for data-heavy list components.

Signals — the future (Angular 16+)

Angular 16 introduced Signals — a new reactive primitive. A signal is like a box that holds a value and tells Angular exactly which templates to update when it changes. No Zone.js needed.

const count = signal(0);       // create signal
count.set(1);                  // update it
computed(() => count() * 2);  // derived value

The AuthService currently uses BehaviorSubject. In a future refactor this could become a signal.

🎤 Interview Q&A
Q: What is change detection in Angular?
A: Change detection is the process Angular uses to synchronize the component's data model with the DOM. The default strategy uses Zone.js to detect any async activity and then walks the component tree checking for changes. OnPush is an optimization where Angular only checks a component when its inputs change, reducing the number of checks. Angular 16+ introduces Signals as a more granular, Zone-free alternative.

RxJS in This Project

RxJS is a library for handling asynchronous data using Observables. Angular's HttpClient, Router, and Forms all use it heavily.

Key RxJS pieces used here

Operator / ClassUsed whereWhat it does
BehaviorSubjectAuthService isAuthenticated$An Observable that remembers its last value and immediately gives it to new subscribers
tapAfter login/refreshSide effect — do something (store tokens) without changing the value flowing through
catchErrorInterceptors, AuthServiceHandle errors in an Observable stream — return a new Observable or rethrow
switchMapjwtInterceptor retryCancel the current inner Observable and switch to a new one — perfect for retry logic
throwErrorError propagationCreates an Observable that immediately errors — used to pass errors to subscribers
pipeEverywhereChains operators together in sequence

BehaviorSubject vs Observable

// BehaviorSubject — holds current value, emits to new subscribers immediately
private _isAuthenticated = new BehaviorSubject<boolean>(this.hasValidToken());

// Expose as read-only Observable — components can subscribe but not .next()
readonly isAuthenticated$ = this._isAuthenticated.asObservable();

// Update from inside the service only
this._isAuthenticated.next(true);
🎤 Interview Q&A
Q: What is the difference between Subject and BehaviorSubject?
A: A plain Subject only emits to subscribers that were subscribed before the emit. A BehaviorSubject stores the last emitted value and immediately gives it to any new subscriber. For auth state, we use BehaviorSubject so that any component that subscribes to isAuthenticated$ — even after login already happened — immediately gets the correct current value.

Q: What does switchMap do and why use it for token refresh?
A: switchMap cancels the current inner subscription and switches to a new Observable. In the refresh flow: when a 401 comes in, we call refreshToken() which returns an Observable. We then switchMap into a retry of the original request using the new token. If another 401 came in before the refresh finished, switchMap would cancel the previous refresh and start a new one — preventing duplicate refresh calls.
