01 Project Structure & Organisation

Clean Architecture splits a solution into four concentric layers. Each layer has a single, explicit responsibility and a strict rule about which other layers it may reference. Dependencies always point inward — toward the domain — never outward.

Dependencies point INWARD ↓
🌐  API Layer — Controllers, Filters, Middleware, Program.cs
↓ depends on
🗄  Infrastructure Layer — EF Core, Repositories, External Services
↓ depends on
⚙️  Application Layer — Use Cases, Commands, Queries, Interfaces
↓ depends on
💎  Domain Layer — Entities, Value Objects, Domain Events, Rules

What Each Layer Owns (and Must Never Do)

LayerOwnsMust Never…
Domain Entities, value objects, business rules, domain events, custom exceptions Reference EF Core, ASP.NET, MediatR, or any NuGet package
Application Use-case handlers (Commands/Queries), DTO mappings, interface contracts, validation rules Reference EF Core directly, know about HTTP, touch the file system
Infrastructure DbContext, EF migrations, repository implementations, third-party API clients, Azure storage Contain business logic — it's plumbing only
API Controllers, middleware, filters, Program.cs, request/response shaping Call repositories or DbContext directly; run business logic

Restaurant kitchen brigade analogy: The Domain is the head chef's recipe book — pure culinary knowledge, no awareness of suppliers or diners. The Application is the expediter station — it orchestrates which recipes to run in response to a ticket. Infrastructure is the walk-in fridge and the delivery dock — raw ingredients and external suppliers. The API is the front-of-house — it takes orders from diners (HTTP clients) and hands tickets to the expediter. The front-of-house never goes into the walk-in fridge directly; it always goes through the expediter.

Interview Q&A

Q: Why can't the Application layer reference EF Core directly?
A: It would couple your business logic to a specific persistence technology. If you ever switch from SQL Server to Cosmos DB, you'd have to rewrite use-case code. By defining IRepository<T> in Application and implementing it in Infrastructure, you can swap EF for anything — the use cases never change.
Q: Why does Program.cs reference Infrastructure even though the API layer "shouldn't"?
A: Program.cs is the DI composition root — the one place in the entire application that is allowed to know about concrete implementations. It wires interfaces (defined in Application) to implementations (defined in Infrastructure). This is the Dependency Inversion Principle in action. The controllers and handlers themselves never see Infrastructure types.
Q: What goes in a Domain Entity vs a DTO?
A: An Entity encapsulates business state and behaviour — it has identity, enforces invariants, and raises domain events. A DTO is a dumb data-transfer bag shaped to what a particular API consumer needs — no logic, no identity. Mapping happens in the Application layer (via AutoMapper or manual projection).
Q: How do you enforce that lower layers never depend on higher ones?
A: Via ProjectReference direction in .csproj files: Domain.csproj has zero references. Application.csproj references only Domain. Infrastructure.csproj references Domain and Application. API.csproj references Application and Infrastructure. A CI lint step (or NDepend / architecture tests using NetArchTest) can fail the build if any reference points the wrong way.

02 Security Middleware

In ASP.NET Core, the middleware pipeline is a chain of responsibility. Each piece of middleware can inspect the request, pass it to the next handler, and post-process the response. Order is critical — middleware registered earlier wraps everything that follows it.

Canonical Security Middleware Order

C# — Program.csvar app = builder.Build();

// ① Redirect HTTP → HTTPS before anything else sees the request
app.UseHttpsRedirection();

// ② HSTS header — tells browsers to only use HTTPS for N days
if (!app.Environment.IsDevelopment())
    app.UseHsts();

// ③ CORS — must be before routing so preflight OPTIONS requests are handled
app.UseCors("DefaultPolicy");

// ④ Rate limiting — block abusive clients before they hit auth/routing
app.UseRateLimiter();

// ⑤ Request logging / correlation ID injection (custom middleware)
app.UseMiddleware<CorrelationIdMiddleware>();
app.UseMiddleware<RequestLoggingMiddleware>();

// ⑥ Routing — must come before auth so route metadata is available
app.UseRouting();

// ⑦ Authentication — WHO is calling? (validates the JWT)
app.UseAuthentication();

// ⑧ Authorization — are they ALLOWED to? (checks [Authorize] policies)
app.UseAuthorization();

// ⑨ Global exception handler (custom middleware) — must wrap endpoints
app.UseMiddleware<GlobalExceptionMiddleware>();

// ⑩ Map endpoints last
app.MapControllers();
app.MapHealthChecks("/health");

What Breaks if Order is Wrong

MistakeConsequence
UseAuthorization() before UseAuthentication()HttpContext.User is anonymous — every [Authorize] endpoint returns 401 even for valid tokens
UseCors() after UseRouting()CORS headers not set on preflight responses — browsers block the request with a CORS error
UseRateLimiter() after UseAuthentication()Auth code runs for every request before it's throttled — DDoS or credential-stuffing attack hammers your JWT validation logic
Exception handler middleware inside UseRouting()/MapControllers()Unhandled exceptions in endpoint execution leak stack traces to the client instead of being caught and shaped as RFC 7807 ProblemDetails
UseHttpsRedirection() after business middlewareBusiness middleware runs on plain-text HTTP requests before the redirect — potential data interception window

Airport security analogy: The middleware pipeline is like airport security lanes. HTTPS redirect is the entrance check — if you're not using the right door, you're turned around immediately. Rate limiting is the queue management system that stops one person from monopolising all lanes. Authentication is passport control — "who are you?" Authorization is the boarding gate check — "you have a passport, but is it for this flight?" Letting someone skip passport control and go straight to the gate breaks everything.

Interview Q&A

Q: What is the difference between Authentication and Authorization?
A: Authentication verifies identity — it answers "who are you?" by validating credentials (e.g., a JWT token signature and claims). Authorization verifies permission — it answers "are you allowed to do this?" by checking roles or policies against the authenticated identity. You must authenticate before you can authorize.
Q: Why should you never use a wildcard (*) CORS origin in production?
A: A wildcard allows any website to make cross-origin requests to your API with the user's browser cookies and credentials. This enables Cross-Site Request Forgery (CSRF) attacks — a malicious site could silently make authenticated API calls on behalf of a logged-in user. Always enumerate exact allowed origins.
Q: How does rate limiting protect the API?
A: It caps how many requests a client (by IP, API key, or user ID) can make within a time window. This prevents brute-force login attacks, credential stuffing, and resource exhaustion DoS attacks. In .NET 7+ the built-in RateLimiter middleware supports fixed window, sliding window, token bucket, and concurrency policies.
Q: What is HSTS and why is it separate from HTTPS redirection?
A: HTTPS redirection is a server-side redirect — the server responds to HTTP with 301/307 to HTTPS. HSTS (HTTP Strict Transport Security) is a browser-side instruction — once seen, the browser will refuse to make plain HTTP connections to the domain at all, for up to N days. It prevents SSL-stripping attacks where a MITM intercepts the first redirect.

03 Logging

Production logging answers three questions: what happened, when, and which request caused it. Structured logging (outputting JSON instead of plain text) makes logs machine-queryable — you can filter by UserId, CaseId, or HTTP status code in tools like Seq, Azure Monitor, or Kibana.

Serilog Setup (the .NET standard)

C# — Program.cs bootstrap// Create the logger BEFORE the host builds
// so startup errors are captured too
Log.Logger = new LoggerConfiguration()
    .ReadFrom.Configuration(configuration)       // reads from appsettings.json
    .Enrich.FromLogContext()                      // picks up LogContext.PushProperty()
    .Enrich.WithMachineName()
    .Enrich.WithEnvironmentName()
    .WriteTo.Console(new RenderedCompactJsonFormatter())
    .WriteTo.File(
        new JsonFormatter(),
        "logs/app-.json",
        rollingInterval: RollingInterval.Day)
    .WriteTo.ApplicationInsights(                 // Azure telemetry
        telemetryClient, TelemetryConverter.Traces)
    .CreateLogger();

try
{
    Log.Information("Starting host");
    builder.Host.UseSerilog();   // replaces built-in ILogger
    var app = builder.Build();
    // ... middleware + run
}
catch (Exception ex)
{
    Log.Fatal(ex, "Host terminated unexpectedly");
}
finally { Log.CloseAndFlush(); }

Correlation IDs — Tracing a Request End-to-End

C# — CorrelationIdMiddleware.cspublic class CorrelationIdMiddleware
{
    private const string HeaderName = "X-Correlation-ID";
    private readonly RequestDelegate _next;

    public CorrelationIdMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext ctx)
    {
        // Reuse caller's ID or generate a fresh one
        var correlationId = ctx.Request.Headers[HeaderName]
            .FirstOrDefault() ?? Guid.NewGuid().ToString("N");

        // Inject into Serilog's LogContext so every log in this request
        // automatically carries the CorrelationId property
        using (LogContext.PushProperty("CorrelationId", correlationId))
        {
            ctx.Response.Headers[HeaderName] = correlationId;
            await _next(ctx);
        }
    }
}

What NOT to Log (HIPAA / security)

  • Patient names, dates of birth, SSN, insurance numbers (PHI)
  • JWT tokens, passwords, API keys
  • SAS URLs (contain embedded credentials)
  • Full request/response bodies in production

Log entity IDs only: "Patient {PatientId} updated" not "Patient John Smith updated".

Black box flight recorder analogy: A correlation ID is like a flight's tail number — every single event (takeoff thrust, altitude changes, conversations with ATC) is tagged with it. After an incident, investigators can pull all events for tail number N12345 from multiple systems and reconstruct exactly what happened, in order. Without a correlation ID, your logs from the API, the background job, and the database are three separate narratives with no way to join them.

Interview Q&A

Q: What is structured logging and why is it better than string concatenation?
A: Structured logging emits log entries as key-value pairs (JSON) rather than a flat string. _logger.Information("Case {CaseId} created", caseId) stores CaseId as a queryable field. In contrast, "Case " + caseId + " created" buries the value in a string — you can only text-search it, not filter, aggregate, or alert on it. Structured logs enable queries like "show me all errors for CaseId 42 in the last hour."
Q: Where in the middleware pipeline should request logging sit?
A: After the correlation-ID middleware (so the log entries carry the ID) but before authentication — so you capture even failed/rejected requests. Serilog's UseSerilogRequestLogging() adds a single summary log at the end of each request with method, path, status code, and duration — far less noise than logging every step.
Q: How do you prevent log flooding in production?
A: Set minimum level overrides per namespace in appsettings.json — e.g., set Microsoft.EntityFrameworkCore to Warning so EF doesn't log every SQL query. Use log sampling for high-volume endpoints (log 1 in 100 successful health-check hits). Use destructuring policies to cap the size of logged objects.

04 Why Clean Architecture

Clean Architecture is not about layers for the sake of layers — it's about protecting what changes slowly (business rules) from what changes fast (technology choices). It operationalises the Dependency Inversion Principle: high-level policy should not depend on low-level detail; both should depend on abstractions.

Clean Architecture vs Traditional Layered Architecture

ConcernTraditional LayersClean Architecture
Dependency directionUI → Business → Data — flows one way, but dependencies cascade downward and upwardAll layers depend on Domain; Domain depends on nothing
TestabilityHard: must mock or spin up the DB to unit-test business logicEasy: Application layer mocks IRepository — no EF Core needed
Database swapPainful — SQL is woven into the business layerImplement a new IRepository — Application/Domain unchanged
Framework couplingBusiness classes often inherit from ORM base classes or controllersDomain entities are plain C# — no framework inheritance
Team scalingMerge conflicts across single-layer filesFeature-slice folders; teams rarely touch the same file

The Dependency Inversion Principle in Practice

C# — Without DIP (tightly coupled)// BAD: Application directly depends on EF Core
public class CreateCaseHandler
{
    private readonly AppDbContext _db;  // ← Infrastructure leaking into Application!

    public async Task Handle(CreateCaseCommand cmd)
        => _db.Cases.Add(new Case(cmd));
}
C# — With DIP (correctly inverted)// Application defines the abstraction it needs
public interface ICaseRepository           // lives in Application
{
    Task AddAsync(Case c, CancellationToken ct);
}

// Application handler depends ONLY on the interface
public class CreateCaseHandler
{
    private readonly ICaseRepository _repo;  // ← abstraction only

    public async Task Handle(CreateCaseCommand cmd, CancellationToken ct)
        => await _repo.AddAsync(new Case(cmd), ct);
}

// Infrastructure provides the concrete implementation
public class EfCaseRepository : ICaseRepository   // lives in Infrastructure
{
    private readonly AppDbContext _db;
    public async Task AddAsync(Case c, CancellationToken ct)
    {
        await _db.Cases.AddAsync(c, ct);
        await _db.SaveChangesAsync(ct);
    }
}

USB port analogy: Your laptop doesn't know or care whether the device you plug in is made by Logitech or Microsoft — it only knows the USB interface standard. The Dependency Inversion Principle is the USB standard: Application code (the laptop) defines the port shape (ICaseRepository); any infrastructure (any vendor's device) that conforms to that shape can be plugged in without changing the laptop. Switching from EF Core to Dapper is just plugging in a different device — the laptop's logic is untouched.

Interview Q&A

Q: What problem does Clean Architecture actually solve that regular MVC doesn't?
A: In MVC, business logic tends to drift into controllers or service classes that directly call EF Core — making it impossible to unit-test without a database. Clean Architecture enforces a testable boundary: the Application layer's handlers depend only on interfaces, so a unit test can inject a mock repo and verify business logic in milliseconds, with no database, no HTTP stack, no mocks of ASP.NET.
Q: What is the Dependency Inversion Principle (the D in SOLID)?
A: High-level modules (business use cases) should not depend on low-level modules (databases, file systems). Both should depend on abstractions. And abstractions should not depend on details — details should depend on abstractions. Practically: define interfaces in the layer that needs them, implement them in the layer below.
Q: Is Clean Architecture overkill for a small project?
A: Honestly, yes — for a prototype or a 5-endpoint CRUD service it adds ceremony. The payoff comes when: (a) the domain is complex enough to need rich entity behaviour; (b) you need high test coverage without a live database; (c) you have multiple teams working on different features; or (d) the technology choices may change (e.g., moving from SQL Server to Cosmos DB). For HCMS these all apply — HIPAA compliance demands rigorous testing, and Azure technology stack evolution is expected.

Example: Full Command + Handler + Validator

C# — CreateCaseCommand.cs// The message / input DTO
public record CreateCaseCommand
(
    Guid   PatientId,
    string ChiefComplaint,
    Guid   AssignedCoordinatorId
) : IRequest<Guid>;      // returns the new Case's ID

// FluentValidation rules — registered as a pipeline behaviour
public class CreateCaseCommandValidator : AbstractValidator<CreateCaseCommand>
{
    public CreateCaseCommandValidator()
    {
        RuleFor(x => x.PatientId)         .NotEmpty();
        RuleFor(x => x.ChiefComplaint)    .NotEmpty().MaximumLength(500);
        RuleFor(x => x.AssignedCoordinatorId).NotEmpty();
    }
}

// The handler — the actual business logic
public class CreateCaseCommandHandler
    : IRequestHandler<CreateCaseCommand, Guid>
{
    private readonly ICaseRepository   _cases;
    private readonly IPatientRepository _patients;
    private readonly IAuditService      _audit;

    public CreateCaseCommandHandler(
        ICaseRepository cases,
        IPatientRepository patients,
        IAuditService audit)
    {
        _cases    = cases;
        _patients = patients;
        _audit    = audit;
    }

    public async Task<Guid> Handle(CreateCaseCommand cmd, CancellationToken ct)
    {
        var patient = await _patients.GetByIdAsync(cmd.PatientId, ct)
            ?? throw new NotFoundException("Patient", cmd.PatientId);

        var newCase = new Case(patient, cmd.ChiefComplaint,
                                   cmd.AssignedCoordinatorId);

        await _cases.AddAsync(newCase, ct);
        await _audit.LogAsync("CaseCreated", newCase.Id, ct);
        return newCase.Id;
    }
}

Pipeline Behaviours — Cross-Cutting Concerns

C# — ValidationBehaviour.cs// Wraps every command/query — runs validators BEFORE the handler
public class ValidationBehaviour<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IBaseRequest
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        if (!_validators.Any()) return await next();

        var context  = new ValidationContext<TRequest>(request);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(f => f != null)
            .ToList();

        if (failures.Any())
            throw new ValidationException(failures);

        return await next();   // call the actual handler
    }
}

Controller — Thin by Design

C# — CasesController.cs[ApiController]
[Route("api/cases")]
[Authorize(Policy = "CareCoordinator")]
public class CasesController : ApiControllerBase
{
    // ISender is MediatR's lightweight interface — only .Send()
    public CasesController(ISender mediator) : base(mediator) { }

    [HttpPost]
    public async Task<IActionResult> Create(CreateCaseCommand cmd)
    {
        // Controller does NOTHING except dispatch the command
        var id = await Mediator.Send(cmd);
        return CreatedAtAction(nameof(GetById), new { id }, new { id });
    }

    [HttpGet("{id:guid}")]
    public async Task<IActionResult> GetById(Guid id)
        => Ok(await Mediator.Send(new GetCaseByIdQuery(id)));
}

Air traffic controller analogy: Pilots (controllers) don't communicate directly with each other — they all talk to the ATC tower (the Mediator). The tower knows which runway handler (handler) to route each flight (command/query) to. If you add a new airline, no existing pilot needs to know about it — they still just talk to the tower. This is why adding a new feature (a new Command + Handler) doesn't require touching the controller at all.

Interview Q&A

Q: What is CQRS and what problem does it solve?
A: Command Query Responsibility Segregation separates write operations (commands) from read operations (queries) into distinct models. The problem it solves: read and write workloads have different needs — reads want denormalized, fast projections; writes want transactional consistency. With CQRS you can optimize each independently: commands use full domain entities with validation; queries can use raw SQL projections, read replicas, or cached views without polluting the write model.
Q: What are MediatR pipeline behaviours and what goes in them?
A: A pipeline behaviour is middleware that wraps every handler — the MediatR equivalent of ASP.NET middleware but at the use-case level. Common behaviours: ValidationBehaviour (runs FluentValidation before the handler; throws if invalid), LoggingBehaviour (logs the command name, duration, and outcome), TransactionBehaviour (wraps commands in a DB transaction), ExceptionBehaviour (catches domain exceptions and converts them to typed errors). Each behaviour calls next() to proceed to the next layer.
Q: Why use ISender instead of IMediator in controllers?
A: ISender exposes only Send(), which is all a controller ever needs. IMediator also exposes Publish() (for notifications) and the full interface. Depending on the narrowest interface is a best practice (Interface Segregation Principle) — it also makes mocking in tests cleaner.
Q: How do you unit test a MediatR handler?
A: Directly — no web host needed. Construct the handler with mocked dependencies (using Moq or NSubstitute), call Handle(command, CancellationToken.None), and assert on the return value and on mock method calls. The handler is a plain class — you don't need to spin up the mediator, the pipeline, or the HTTP stack.

06 Host Configuration

In .NET 6+, Program.cs is the single bootstrap file that wires together the DI container and the middleware pipeline. Understanding it means understanding how the application comes alive from nothing to serving requests.

Captive Dependency Bug — The Most Common DI Mistake

If you inject a Scoped service into a Singleton, the scoped service is captured at singleton creation time and reused across all requests — meaning it carries state (like DbContext) from a previous user's request into the next one. This causes data leaks and concurrency bugs. ASP.NET will throw InvalidOperationException at startup if scope validation is enabled (it is, in Development).

Extension Methods — Keeping Program.cs Clean

C# — Application/DependencyInjection.cspublic static class ApplicationServiceExtensions
{
    // Registers all Application-layer services
    public static IServiceCollection AddApplication(
        this IServiceCollection services)
    {
        // MediatR scans the assembly for all IRequestHandler implementations
        services.AddMediatR(cfg =>
            cfg.RegisterServicesFromAssembly(
                Assembly.GetExecutingAssembly()));

        // FluentValidation: registers all validators in this assembly
        services.AddValidatorsFromAssembly(Assembly.GetExecutingAssembly());

        // Register pipeline behaviours in order of execution
        services.AddTransient(typeof(IPipelineBehavior<,>),
                              typeof(LoggingBehaviour<,>));
        services.AddTransient(typeof(IPipelineBehavior<,>),
                              typeof(ValidationBehaviour<,>));

        // AutoMapper: scans for Profile classes
        services.AddAutoMapper(Assembly.GetExecutingAssembly());

        return services;
    }
}

Restaurant opening sequence analogy: WebApplication.CreateBuilder(args) is like unlocking the building and turning on the lights — basic infrastructure is ready. Service registration (Phase 2) is mise en place — stocking every station with what it needs before service starts. builder.Build() is the head chef calling "kitchen ready" — no more restocking after this point. The middleware pipeline (Phase 4) is the order in which stations process each ticket: the host stand (routing), the pass (auth), then the kitchen stations (handlers). app.RunAsync() is opening the doors to customers.

Interview Q&A

Q: What does WebApplication.CreateBuilder() actually do under the hood?
A: It creates a WebApplicationBuilder that pre-wires: (1) IConfiguration with the default layered sources (appsettings.json → env → command-line); (2) Kestrel as the HTTP server; (3) the IHostEnvironment from the ASPNETCORE_ENVIRONMENT env var; (4) default logging providers (Console, Debug, EventSource); and (5) an empty DI IServiceCollection ready to be populated. It does NOT start the server — that happens at app.RunAsync().
Q: What is the difference between Scoped and Transient service lifetimes?
A: Scoped creates one instance per HTTP request — all services within that request share the same instance (useful for DbContext, which tracks entities per-unit-of-work). Transient creates a new instance every time the service is resolved — safe for lightweight, stateless helpers. The key question: "does this service hold state I want to share within a request, or state I definitely don't want to share?"
Q: Why are secrets not stored in appsettings.json?
A: appsettings.json is committed to source control. A secret in source control is compromised the moment the repository is cloned by anyone — or leaked via a public GitHub repo. Secrets go in user-secrets (local dev — stored in the OS user profile, never in the repo), environment variables (CI/CD pipelines), or Azure Key Vault (production — accessed via Managed Identity, no stored credentials at all).
Q: How do you run EF Core migrations at startup in production safely?
A: Resolve AppDbContext from the DI container before app.Run() and call context.Database.MigrateAsync(). However, for production this is often undesirable — it blocks startup, requires elevated DB permissions on the service account, and is hard to roll back. The safer pattern is to run migrations as a separate job in the CI/CD pipeline (using dotnet ef database update) before deploying the new binary, and keep the service account read/write but not schema-owner.
Q: How does the configuration system support multiple environments without code changes?
A: By setting ASPNETCORE_ENVIRONMENT to Development, Staging, or Production, .NET automatically loads appsettings.{Environment}.json on top of the base appsettings.json. Values in the environment-specific file override the base. This means you ship one binary and control its behaviour entirely through config — the 12-Factor App principle. Secrets layer on top via environment variables or Key Vault, also without touching the binary.
