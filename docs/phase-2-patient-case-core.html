Executive Summary

Phase 2 delivered the core patient and case management lifecycle — the beating heart of HCMS. Before this phase, the app could authenticate users. After it, care coordinators can register patients, open cases, progress them through a validated status workflow, build care teams, add clinical notes, and have every data change automatically recorded in an immutable audit trail.

Simultaneously, the Angular frontend grew from a login screen into a fully navigable clinical workspace: paginated patient search, rich case detail with five tabs, role-gated actions, and reusable shared components.

Architecture

HCMS follows Clean Architecture — dependency arrows point inward. No outer layer can break the inner layer's contracts.

Analogy — Hospital departments: Think of the Domain layer as the hospital's core medical procedures (the rules never change regardless of who's funding the hospital). Application is the nursing staff that orchestrates those procedures. Infrastructure is the physical equipment (MRI machines, EHR terminals). The API is the reception desk — it talks to patients but delegates everything real to the staff inside.

Dependency arrows always point inward. Domain has zero external dependencies.

DataHandler Pattern

Every feature area has a matched interface + implementation pair. Controllers stay thin — they only deserialize the request and call the handler.

Interface (Application layer) 
public interface IPatientDataHandler
{
    Task<PatientDto> GetByIdAsync(Guid id);
    Task<PatientDto> CreateAsync(CreatePatientRequest r);
    // …
}
Controller (API layer) 
[HttpPost]
public async Task<IActionResult> Create(
    [FromBody] CreatePatientRequest r)
{
    var result = await _handler.CreateAsync(r);
    return CreatedAtAction(nameof(GetById),
        new { id = result.Id }, result);
}

Backend Work

Audit Interceptor (EF Core SaveChanges Override)

Every create, update, or delete on a tracked entity automatically writes a row to the AuditLogs table — no developer needs to remember to call a logging method.

Analogy — Security camera system: Instead of asking nurses to fill out a paper log every time they enter a medication room, you install a camera that captures every entry automatically. The staff can't forget, can't skip it, and can't tamper with the recording. The AuditInterceptor is that camera — wired directly into EF Core's save pipeline so no business code has to think about it.

Two-phase save: entity data first, then audit rows. Prevents infinite recursion because AuditLog itself is excluded from the audited types set.

Implementation Details

Audited entity types (compile-time HashSet<Type>): Patient, Case, CaseNote, User

AuditLog has no FK to Users — this is a HIPAA hard rule. A deleted user's audit history must survive intact.

// ApplicationDbContext.cs — SaveChangesAsync override
public override async Task<int> SaveChangesAsync(CancellationToken ct = default)
{
    var auditEntries = BuildAuditEntries();        // ① snapshot before save
    var result = await base.SaveChangesAsync(ct);  // ② save real entities
    if (auditEntries.Count > 0)
    {
        AuditLogs.AddRange(auditEntries);
        await base.SaveChangesAsync(ct);           // ③ save audit rows
    }
    return result;
}

nullable IHttpContextAccessor: Added as an optional constructor parameter so the design-time factory (ApplicationDbContextFactory) can still call new ApplicationDbContext(options) with one argument — critical for dotnet ef migrations add to work without a running DI container.

UsersController & UserDataHandler

Admin-only endpoint surface for user management. All endpoints decorated with [Authorize(Roles = "Admin")] at the controller class level.

Analogy — Hospital HR department: Only the HR manager (Admin) can hire new staff, assign departments (roles), or suspend employee access. Individual clinicians cannot see or change other people's accounts.

Duplicate validation: before creating, the handler queries for matching Username or Email and throws ValidationException if found — never leaks which one matched (prevents username enumeration).

Role assignment: atomically removes all existing UserRoles rows and re-inserts the new set within the same SaveChangesAsync call, so roles can never be in a half-assigned state.

CaseNoteEditLockService (Background Job)

A hosted service that runs every hour, scanning for case notes created more than 24 hours ago that are still editable, and locking them.

Analogy — Medical chart counter-signature window: In hospitals, a nurse's note can be amended for 24 hours after it's written, but after that the chart is considered final and tamper-evident. This background job is the supervisor who walks through at shift change and stamps those charts "No Further Edits."
Implementation Highlights 
// Why IServiceScopeFactory and not direct DbContext injection?
// BackgroundService lifetime = Singleton.
// ApplicationDbContext lifetime = Scoped.
// Injecting scoped into singleton = captive dependency error.
// Solution: create a fresh scope per run, resolve DbContext inside it.

protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        await Task.Delay(TimeSpan.FromHours(1), stoppingToken);
        using var scope = _scopeFactory.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
        var cutoff = DateTime.UtcNow.AddHours(-24);
        await db.CaseNotes
            .Where(n => n.IsEditable && !n.IsDeleted && n.CreatedAt <= cutoff)
            .ExecuteUpdateAsync(s =>
                s.SetProperty(n => n.IsEditable, false), stoppingToken);
    }
}

ExecuteUpdateAsync (EF Core 7+) issues a single bulk UPDATE statement — no entity tracking overhead, no round-trips per note.

Case Status State Machine

Status transitions are validated server-side in CaseDataHandler and mirrored client-side in the TRANSITIONS map in CaseDetailComponent. The UI can only offer valid next states; the API will reject anything else.

Key Decisions & Trade-offs

Decision 1 — SaveChanges override vs. ISaveChangesInterceptor

Chosen: override SaveChangesAsync directly in ApplicationDbContext.

Rejected: EF Core's ISaveChangesInterceptor registration via AddInterceptors().

Why: The override keeps the logic co-located with the context, making it obvious to any developer reading the file. The interceptor API is more powerful but adds indirection (a separate class registered through DI) that isn't needed for a project of this size. The override is also easier to test by subclassing ApplicationDbContext.

Decision 2 — Nullable IHttpContextAccessor in DbContext constructor

Problem: Adding IHttpContextAccessor to the constructor breaks the design-time factory used by dotnet ef migrations add — the factory creates the DbContext with only the DbContextOptions argument.

Solution: Make the parameter optional: IHttpContextAccessor? httpContextAccessor = null. The factory continues to work; the runtime DI container wires it up when the app is running.

Trade-off: Null-checking inside BuildAuditEntries(). Acceptable — audit entries fall back to no ChangedBy user ID when the context is null (migrations, seed, tests).

Decision 3 — IServiceScopeFactory in BackgroundService

Problem: BackgroundService (registered as Singleton) cannot directly inject ApplicationDbContext (Scoped). Doing so creates a "captive dependency" — the singleton holds onto a scoped instance for the entire app lifetime, causing stale DbContext and connection leaks.

Solution: Inject IServiceScopeFactory (itself a Singleton). Each run of the job creates a short-lived scope, resolves ApplicationDbContext within it, and disposes the scope when done. Exactly one DB connection per hour-long cycle.

Decision 4 — Client-side TRANSITIONS map mirrors server-side

Why duplicate the logic? The server validates every transition and will reject an invalid one with HTTP 422. But if the UI only shows valid transitions, users never reach an error state — better UX. The two copies are intentionally redundant; the server copy is the source of truth.

Risk: If the state machine changes on the server, the UI map must be updated manually. Acceptable for a portfolio project; in a real system this could be driven by a GET /api/cases/transitions endpoint.

Decision 5 — AuditLog has no FK to Users

HIPAA/compliance hard rule: Audit records must survive user deletion. A FK would either cascade-delete audit rows (data loss) or block user deletion (broken UX). Instead, ChangedBy stores the user's ID as a plain string. Historical lookups join on the string if the user still exists; old records stay readable even if the user is gone.
